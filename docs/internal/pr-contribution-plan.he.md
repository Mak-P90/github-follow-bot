# Pull Requests Plan עבור תרומות

מסמך זה מגדיר את **המדריך החובה המינימלי** לכל מפתח להציע שינויים על ידי Pull Request (PR) באופן עקבי, ניתן לביקורת ומיושר עם הכללים של מאגר זה.

## זרימת סיכום

1. הכן סניף מ-`main` עם שם ברור (`feat/...`, `fix/...`, `docs/...`).
2. השלם את תבנית יחסי הציבור בעקבות הסעיפים של תוכנית זו.
3. בצע אימות חובה מינימלי (בהתאם לשער החליפין).
4. יחסי ציבור פתוחים עם הוכחות טכניות ורשימת תאימות.
5. השתתף בסקירה מבלי לאבד את העקיבות (הערות, החלטות ושינויים).

---

## מבנה יחסי ציבור חובה

### 1) דוק

כולל תיעוד ברור כדי שהבודקים והמפעילים יבינו את השינוי:

- **סיכום קצר** (מה משתנה ולמה).
- **היקף** (מודולים/קבצים מושפעים).
- **השפעה תפעולית** (אם תשנה ביצוע, אישור, תורים, התמדה או אבטחה).
- **תיעוד מעודכן**:
    - `README.md` כאשר התנהגות גלויה משתנה.
    - `ENTERPRISE_HARDENING_ASSESSMENT.md` כאשר מוחל על בקרות ארגוניות/הקשחה.

### 2) בעיה

תאר את בעיית השורש באמצעות ראיות:

- סימפטום ניתן לצפייה.
- הקשר (מתי זה מתרחש, מי זה משפיע, חומרה).
- ראיות ניתנות לשחזור (יומנים, פלט פקודה, בדיקות, בעיה מקושרת).
- סיכון של אי תיקון.

### 3) פתרון

הסבר את האסטרטגיה המיושמת:

- החלטה טכנית עיקרית וחלופות שנמחקו.
- שינויים שבוצעו בכל שכבה (תצורה, תחום, מתאמים, התמדה, CLI, מסמכים).
- תאימות לאחור ותוכנית הגירה (אם יש שינויים בחוזה).
- סיכונים שיוריים והפחתות.

### 4) עומד בסטנדרטים מינימליים

רשימת בדיקה מינימלית שיש לבדוק לפני בקשת מיזוג:

- [ ] שינוי מחולק באחריות (ללא פונקציות ענק ערבוב רשת + SQL + עסקים).
- [ ] סודות מחוץ לקוד ויומנים מחוטאים.
- [ ] אימפוטנציה נשמרת (אם אתה משחק follow/colas).
- [ ] ראיות אימות הכלולות ב-PR.
- [ ] תיעוד מעודכן (`README.md` ו/או הערכה אם רלוונטי).
- [ ] חפצים מקומיים ללא גרסה (`.db`, יומנים, `__pycache__` וכו').

**אם קוד Python שונה**, צרף פלט של:

- `python -m py_compile bot.py check_all_followers.py`
- `pytest -q`
- `python bot.py doctor`

### 5) תקנים אופציונליים (מומלץ)

הם לא חוסמים בעצמם, אבל הם משפרים את איכות יחסי הציבור:

- [ ] הוסף/עדכן ADR אם ארכיטקטורה או חוזים רלוונטיים משתנים.
- [ ] כלול תוכנית החזרה מפורשת.
- [ ] צרף דוגמאות לשימוש לפני/אחרי CLI.
- [ ] הוסף מדדים צפויים (השהייה, נפח, שגיאות) כדי להקל על צפיות.
- [ ] הצע משימות עתידיות (חוב טכנולוגי) בהיקף מוגבל.

### 6) תקנים שלא עומדים על ידי...

אם משהו אינו נכון, יש להצהיר עליו **מפורש**:

- תקן לא עומד.
- סיבה ספציפית (חוב קודם, תלות חיצונית, מגבלה סביבתית וכו').
- סיכון מונחה.
- תוכנית ומועד סגירה משוער.
- אחראי על ניטור.

> "בהמתנה" לא מתקבל ללא בעלים או תוכנית.

```md
- [ ] Idempotencia preservada
- [ ] Validación ejecutada
- [ ] README/Assessment עודכנו כאשר רלוונטי
- [ ] היגיינת מאגר תואמת

- `python -m py_compile bot.py check_all_followers.py`

- [ ] ADR actualizado
- [ ] תוכנית Rollback
- [ ] Ejemplos CLI antes/después
- [ ] Métricas esperadas

## 6) Normas no cumplidas por...

- Norma:
- Motivo:
- Riesgo:
- תוכנית סגירה:
- Responsable:
```

## קריטריוני קבלה למיזוג

יחסי ציבור מוכנים להתמזג כאשר:

1. הסעיפים 1-6 הושלמו (ללא שדות קריטיים ריקים).
2. אימות מינימלי בוצע או חריגה מוצדקת עם בעל + תוכנית.
3. הוא אינו מציג רגרסיות אבטחה, התמדה או חוזי אישור.
4. שומר על יכולת מעקב מספקת לביקורת הבאה.
