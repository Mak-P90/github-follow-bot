# План запросов на внесение вкладов

Этот документ определяет **минимальное обязательное руководство** для любого разработчика, позволяющее предлагать изменения посредством запроса на включение (PR) последовательным, проверяемым и согласованным образом с правилами этого репозитория.

## Сводный поток

1. Подготовьте ветку из`main`с понятным названием(`feat/...`, `fix/...`, `docs/...`).
2. Заполните шаблон PR, следуя разделам этого плана.
3. Выполнить обязательные минимальные проверки (в зависимости от обменного курса).
4. Открытый PR с техническими доказательствами и контрольным списком соответствия.
5. Присутствовать на проверке, не теряя отслеживаемости (комментариев, решений и изменений).

---

## Обязательная PR-структура

### 1) Док

Включает четкую документацию, позволяющую рецензентам и операторам понять изменения:

- **Краткое описание** (что меняется и почему).
- **Область** (затронутые модули/файлы).
- **Влияние на эксплуатацию** (если вы измените выполнение, аутентификацию, очереди, постоянство или безопасность).
- **Обновленная документация**:
    - `README.md`когда видимые изменения в поведении.
    - `ENTERPRISE_HARDENING_ASSESSMENT.md`применительно к корпоративным/укрепляющим средствам контроля.

### 2) Проблема

Опишите основную проблему с доказательствами:

- Наблюдаемый симптом.
- Контекст (когда это происходит, на кого это влияет, тяжесть).
- Воспроизводимые доказательства (журналы, вывод команд, тесты, связанные проблемы).
- Риск неисправления.

### 3) Решение

Объясните применяемую стратегию:

- Основное техническое решение и отброшенные альтернативы.
- Изменения, внесенные на каждом уровне (конфигурация, домен, адаптеры, постоянство, CLI, документация).
- Обратная совместимость и план миграции (при наличии изменений в контракте).
- Остаточные риски и меры по их смягчению.

### 4) Соблюдаются минимальные стандарты

Минимальный контрольный список, который необходимо проверить перед запросом на слияние:

- [ ] Изменения разделены по ответственности (никаких гигантских функций, смешивающих сеть + SQL + бизнес).
- [ ] Секреты, не входящие в код, и очищенные журналы.
- [ ] Сохранена идемпотентность (если вы играете в Follow/colas).
- [ ] Доказательства валидации включены в PR.
- [ ] Обновлена ​​документация (`README.md`и/или оценку, когда это применимо).
- [ ] Неверсионные локальные артефакты (`.db`, журналы,`__pycache__`, и т. д.).

**Если код Python изменен**, прикрепите вывод:

- `python -m py_compile bot.py check_all_followers.py`
- `pytest -q`
- `python bot.py doctor`

### 5) Дополнительные правила (рекомендуется)

Они не блокируют сами по себе, но улучшают качество пиара:

- [ ] Добавить/обновить ADR, если изменится архитектура или соответствующие контракты.
- [ ] Включить явный план отката.
- [ ] Прикрепите примеры использования CLI до/после.
- [ ] Добавьте ожидаемые метрики (задержка, объем, ошибки) для облегчения наблюдения.
- [ ] Предлагайте будущие задачи (технический долг) с ограниченным объемом.

### 6) Правила не соблюдаются...

Если что-то неверно, это должно быть **явно** объявлено:

- Стандарт не соблюден.
- Конкретная причина (предыдущий долг, внешняя зависимость, экологические ограничения и т. д.).
- Риск предполагается.
- План и предполагаемая дата закрытия.
- Ответственный за мониторинг.

> «В ожидании» не принимается без владельца или плана.

## Критерии приемки слияния

PR готов к слиянию, когда:

1. Разделы 1–6 заполнены (нет пустых критических полей).
2. Минимальные проверки, выполненные или обоснованное исключение с владельцем + планом.
3. Не вносит регресс в безопасность, постоянство или контракты аутентификации.
4. Поддерживает достаточную прослеживаемость для последующего аудита.
