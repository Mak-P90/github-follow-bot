# Правила Pull Request для переводов

Этот раздел определяет минимальные критерии принятия изменений перевода документации.

## 1. Область и структура

- Сохранять ту же структуру файлов, что и в базовой испанской документации.
- Использовать языковые суффиксы на каждой странице (`.en.md`, `.fr.md`, `.he.md`, `.ru.md`, `.zh.md`).
- Не удалять технические разделы исходного документа при переводе.

## 2. Качество контента

- Сохранять технический смысл, команды и примеры конфигурации.
- Не переводить имена переменных, CLI-флаги, названия таблиц и идентификаторы кода.
- Сохранять блоки кода и форматирование Markdown.

## 3. Терминологическая согласованность

- Повторно использовать глоссарий проекта для повторяющихся терминов.
- Если вводится новый термин, документировать его в той же PR.
- Избегать дословных переводов, меняющих операционный контекст.

## 4. Требования к ревью

- Указывать в PR список затронутых страниц по языкам.
- Прикладывать подтверждение успешной локальной сборки MkDocs.
- Запрашивать минимум одно ревью от человека с техническим контекстом затронутого модуля.

## 5. Минимальный PR-checklist

- [ ] Переводные файлы созданы/обновлены с корректным суффиксом.
- [ ] Навигация `mkdocs.yml` проверена для новых языков.
- [ ] Сборка документации выполнена локально.
- [ ] Техническая терминология проверена и согласована.
- [ ] Изменения объёма работ задокументированы (если применимо).
- [ ] Если добавлены операционные команды/флаги/runbook-и, обновлены все доступные языковые варианты в той же PR.
